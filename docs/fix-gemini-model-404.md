# Fix: Gemini model 404 ("gemini-2.5-flash is no longer available")

## Root cause

Tips are generated by Google's Gemini API, and the model name hardcoded in the
code — `gemini-2.5-flash` — has been retired by Google. When the code asks Gemini
for a tip, the API rejects the request with `404 NOT_FOUND` and the message
"This model models/gemini-2.5-flash is no longer available." The service catches
that exception and forwards the raw error text to the phone as the
"⚠️ Health tips problem" notification (see `src/main.py:131-139`).

The single offending line:

- `src/analyze.py:7` — `MODEL = "gemini-2.5-flash"`

That constant is passed straight to `client.models.generate_content(model=MODEL, ...)`
at `src/analyze.py:60-61`. Nothing else references the model name, so this is a
one-line configuration problem, not a logic bug.

## Why it happened

Google periodically deprecates and removes specific dated/versioned Gemini model
IDs. Pinning to a bare `gemini-2.5-flash` string means the app breaks the moment
that exact ID is retired, with no fallback.

---

## Plan for the fix

**Goal:** get tips flowing again and make the model choice resilient so this
doesn't silently recur.

### Step 1 — Confirm the current valid model name (do NOT trust training data)

Model availability changes over time, so the exact replacement ID must be
verified live, not guessed. Run a quick check against the account's own API key:

```bash
cd /Users/unoa/garmin-health-tips
python -c "from google import genai; [print(m.name) for m in genai.Client().models.list()]"
```

(Requires `GEMINI_API_KEY` in the environment, same key the workflow uses.)
Pick a currently-listed flash-tier model from the output.

**Recommendation:** prefer a rolling alias such as `gemini-flash-latest` if it
appears in the list, because it auto-tracks Google's current flash model and
won't 404 the next time a dated version is retired. If only dated IDs are
available (e.g. a newer generation), pick the newest stable flash one. Avoid
`-preview`/experimental IDs for a scheduled production job.

### Step 2 — Update the model constant

In `src/analyze.py:7`, change `MODEL = "gemini-2.5-flash"` to the verified name
from Step 1.

Also confirm the `thinking_config` / `ThinkingConfig(thinking_budget=0)` call at
`src/analyze.py:63-68` is still accepted by the chosen model — the "disable
thinking" option is model-specific. If the new model rejects it, remove that
argument (a dry-run in Step 3 will surface this).

### Step 3 — Verify end-to-end before declaring it fixed

Run the real pipeline without sending a notification:

```bash
python -m src.main --dry-run --slot midday
```

Success = it prints a plausible tip and exits 0. Repeat for `--slot morning`
(exercises the calorie-target path) and `--slot evening`. If any run still
errors, read the actual message rather than assuming.

### Step 4 — (Optional hardening, ask the user first)

Two small, independent improvements that go beyond the minimal fix:

- Move `MODEL` into `config.yaml` so the model can be changed without editing code.
- In `src/analyze.py`, catch a 404 / "no longer available" error and retry once
  against a `-latest` alias, so a future retirement self-heals instead of paging
  the phone.

### Notes for whoever implements it

- No dependency, workflow, or secret changes are needed — `google-genai` and
  `GEMINI_API_KEY` are already wired up correctly (`.github/workflows/`,
  `requirements.txt`).
- This is genuinely a one-line change plus verification. Don't refactor
  `analyze.py` beyond Steps 2–4.
- There are tests in `tests/`; run `python -m pytest` after the change to make
  sure nothing that mocks or references the model broke.
